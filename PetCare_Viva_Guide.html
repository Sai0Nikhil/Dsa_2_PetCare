🌳 M1 – Trees & Balanced Search Structures

What we implemented

  • BST — insert, search, delete, in-order/pre-order/post-order traversals
  • AVL Tree — self-balancing BST using rotations (LL, RR, LR, RL)
  • Height comparison between BST and AVL on the same dataset

Domain use-case

Pet records are indexed by PetID. A plain BST gives O(log n) average but O(n) worst case on sorted insertions. The AVL Tree guarantees O(log n) always by re-balancing after every insert/delete. In-order traversal produces a sorted treatment history.

Q: What is the difference between BST and AVL?
A BST has no balance guarantee — if you insert sorted data it becomes a linked list with O(n) operations. An AVL Tree enforces the balance factor (−1, 0, or +1) at every node using rotations, keeping the height at most 1.44 log n, which guarantees O(log n) for insert, search, and delete.
Q: What are the four AVL rotations?
  • LL rotation (right rotation) — inserted left of left child
  • RR rotation (left rotation) — inserted right of right child
  • LR rotation (left-right) — inserted right of left child: first left-rotate child, then right-rotate root
  • RL rotation (right-left) — inserted left of right child: first right-rotate child, then left-rotate root
Q: What are the three tree traversals and their uses?
  • In-order (Left→Root→Right) — gives sorted output for a BST
  • Pre-order (Root→Left→Right) — useful for copying/serializing the tree
  • Post-order (Left→Right→Root) — useful for deleting the tree

🌲 M2 – Multiway Trees & Range Queries

What we implemented

  • B-Tree (order 3) — multiway search tree for clinic record IDs
  • B+ Tree — range index for vaccination schedules
  • Segment Tree — range sum queries on daily clinic visit counts
  • Fenwick Tree (BIT) — cumulative treatment statistics with prefix sums
Q: Why use a B-Tree instead of an AVL Tree for database indexing?
B-Trees are designed for disk storage. Each node can hold many keys (reducing the number of disk reads). An AVL Tree node holds one key, so for a million records you'd need ~20 disk reads. A B-Tree with order 500 needs only 3–4 disk reads for the same data. B+ Trees also keep all data in leaves linked as a list, making range queries very fast.
Q: What is the difference between B-Tree and B+ Tree?
In a B-Tree, data records are stored in both internal and leaf nodes. In a B+ Tree, internal nodes store only keys for routing, and ALL data records are stored in leaf nodes which are linked together. This makes B+ Trees better for range queries: once you find the start key you just follow the leaf linked list.
Q: How does a Segment Tree work?
A Segment Tree is a binary tree where each node stores the aggregate (sum, min, max) of a range of the array. The root covers the full array, its children cover halves, and so on. A range query is answered in O(log n) by combining at most O(log n) nodes. Point updates also take O(log n).
Q: What is a Fenwick Tree (BIT)?
A Binary Indexed Tree stores partial sums using bit manipulation. Each index i stores the sum of a range of length equal to the lowest set bit of i. Prefix sum and point update are both O(log n), and the space is O(n) — simpler to code than a Segment Tree for pure prefix sum queries.

What we implemented

  • DFS — checks workflow connectivity and counts connected components

Q: What is the difference between BFS and DFS?
BFS explores level by level (nearest nodes first) using a Queue. It finds the shortest path in unweighted graphs. DFS explores as deep as possible first using a Stack (or recursion). BFS is better for shortest paths; DFS is better for detecting cycles, connected components, and topological sort.
Q: What is an MST? Kruskal vs Prim?
A Minimum Spanning Tree connects all vertices with minimum total edge weight and no cycles. Kruskal's sorts all edges by weight and greedily adds edges that don't form a cycle (uses Union-Find). Prim's grows the tree from a starting vertex by always adding the cheapest edge connecting the tree to a new vertex. Both give O(E log E) for sparse graphs.

🗺️ M4 – Shortest Path Algorithms

What we implemented

  • Dijkstra — fastest route from one clinic to all others (positive weights)
  • Bellman-Ford — handles negative referral weights, detects negative cycles
  • Floyd-Warshall — all-pairs shortest distances (full matrix)
  • Topological Sort — schedules treatment procedures (Admit→Diagnose→Surgery→...)
Q: When to use Dijkstra vs Bellman-Ford vs Floyd-Warshall?
  • Dijkstra — single source, all edges positive, O((V+E) log V). Fastest for positive-weight graphs.
  • Bellman-Ford — single source, can handle negative weights, detects negative cycles. O(VE). Slower but more general.
  • Floyd-Warshall — all pairs shortest paths, O(V³). Use when you need distances between every pair of vertices.
Q: What is Topological Sort and where did you use it?
Topological Sort orders vertices of a DAG (Directed Acyclic Graph) so that for every directed edge u→v, u comes before v. We used Kahn's algorithm (BFS-based) to schedule treatment procedures: Admit → Diagnose → X-Ray / Lab Test → Surgery → Recovery → Discharge. It only works on DAGs — if there is a cycle it means a circular dependency.

📊 M5 – Divide & Conquer / Sorting

What we implemented (in scope for review)

  • Merge Sort — sorts treatment records by cost (stable, O(n log n))
  • Quick Sort — ranks diagnoses by frequency (in-place, O(n log n) avg)
  • Heap Sort and Radix/Counting Sort — Coming Soon
Q: Explain Divide & Conquer with Merge Sort.
Divide: Split the array into two halves. Conquer: Recursively sort each half. Combine: Merge the two sorted halves into one. Time: O(n log n) always. Space: O(n) extra. It is stable — equal elements keep their original order.
Q: Merge Sort vs Quick Sort — which is better?
  • Merge Sort: O(n log n) guaranteed, stable, needs O(n) extra space. Better for linked lists and when stability matters.
  • Quick Sort: O(n log n) average, O(n²) worst case, in-place (O(log n) stack). Better for arrays in practice due to cache efficiency.
In PetCare we use Merge Sort for cost-based sorting (stability matters) and Quick Sort for frequency ranking.

📁 CSV Feature (New Addition)

What it does

At startup the user chooses between two modes:

  • Load from CSV — reads data/pets.csv (45 pre-defined pet records) into memory via CsvLoader.java and stores them in AppData.java
  • Continue as Raw — each module generates its own sample data using DataGenerator.java
Q: Why did you add a CSV feature?
To make the demonstration more realistic. Hard-coded random data changes every run, making it hard to trace results. The CSV gives us consistent, predictable pet records with realistic names, species, and breeds so we can show the same data across all modules and explain results clearly.

⚠️ Likely Viva Questions — Quick Answers

Q: Why Java?
Java is object-oriented, strongly typed, and platform-independent. It has built-in generics, collections, and automatic memory management, making it ideal for implementing and demonstrating data structures clearly without manual memory concerns.
Q: What is the time complexity of AVL insertion?
O(log n) — the height of an AVL tree is always O(log n), and at most one rotation is needed after insertion.
Q: What is a connected component?
A maximal subgraph in which every vertex is reachable from every other vertex. We count them using DFS — every time DFS is called on an unvisited vertex, it explores a new component.
Q: How does Kruskal's algorithm use Union-Find?
Union-Find tracks which vertices are in the same component. When Kruskal picks the next cheapest edge (u, v), it calls Find(u) and Find(v) — if they are in the same component, adding the edge would create a cycle, so it's skipped. If they are in different components, the edge is added and Union(u, v) merges the components. This keeps the MST cycle-free.
Q: Can Dijkstra handle negative weights?
No. Dijkstra assumes that once a vertex is finalized, its distance is correct — a negative edge later could give a shorter path and violate this. Use Bellman-Ford for negative weights.
Q: What is the balance factor in AVL?
Balance Factor = Height(left subtree) − Height(right subtree). It must be −1, 0, or +1 at every node. If it becomes +2 or −2, a rotation is performed to restore balance.
Q: What is lazy propagation in Segment Trees?
When we do a range update (e.g., add 5 to all elements from index 2 to 8), instead of updating every affected node immediately, we mark the node with a "lazy tag" and propagate it downward only when that node's children are actually needed. This reduces range update time from O(n) to O(log n).

💡 Tips for the Viva

Always connect the algorithm to the domain. Don't just say "we implemented Dijkstra." Say "we use Dijkstra to find the shortest travel route between any two clinics in our healthcare network, for example from MainVet in Chennai to HappyTails in Pune."
Know your numbers. The graph has 6 clinics and 9 edges. The CSV has 45 pets with IDs 101–145. BST height vs AVL height on 45 records will differ visibly — AVL will be smaller.
Coming Soon is fine — explain why. If asked about M6: "We have the code structure and class stubs ready. The algorithms (Activity Selection, Knapsack, LIS) are planned for the final submission — the menu shows Coming Soon so the reviewer can see we are aware of what remains."
Don't confuse B-Tree and B+ Tree. B-Tree stores data in all nodes. B+ Tree stores data only in leaf nodes (leaves are linked). The B+ Tree is what databases actually use for indexes because its linked leaves make range scans fast.
